CHAPTER 01

資料庫概念 & 系統架構

⚠️傳統檔案系統的問題

  • Data Redundancy（資料重複）：同樣的資料存在多個檔案，浪費空間且容易不一致
  • Data Isolation（資料孤立）：不同程式的檔案格式各異，新程式難以讀取舊資料
  • Concurrent Access 問題：多人同時讀寫同一檔案 → 可能產生 Dirty Reads、Phantom Reads 等
  • Security Problem：程式可任意存取檔案，無法細粒度控制
  • Integrity Problem：難以確保資料正確與一致
  • Data/Program Dependency：資料格式改變時，所有程式都要跟著改

🗄️Database 定義

代表真實世界某部分（Universe of Discourse），由自描述（Self-describing）的相關整合資料集合組成，供多個應用程式共用。

關鍵詞
Self-describing = 包含資料本身 + Metadata（描述結構）
Integrated = 消除重複，統一格式
Shared = 多程式共用

📋Schema vs Instance

Schema（Metadata）：資料庫的「描述」，包括欄位名稱、型別、關係、限制等。不常改變。

Instance（Database State）：某一時刻資料庫中的實際資料。經常改變。

🏛️ANSI/SPARC 三層架構

⭐
這是很重要的考點！三層分別是什麼、各層的 Schema 名稱、以及兩種 Data Independence。
層次名稱說明Schema
External 個人使用者視角 每個使用者/程式看到的部分資料（View） External Schema（可多個）
Conceptual 社群整體視角 整個資料庫的邏輯結構，獨立於儲存細節 Conceptual Schema（只有一個）
Internal 儲存實體視角 實際儲存方式（索引、存放格式等） Internal Schema（只有一個）
Logical Data Independence
改變 Conceptual Schema 而不需要改 External Schema / 應用程式
Physical Data Independence
改變 Internal Schema 而不需要改 Conceptual Schema

⚙️DBMS 功能 & 組件

  • DDL（Data Definition Language）：定義/修改 Schema
  • DML（Data Manipulation Language）：INSERT / UPDATE / DELETE / SELECT
  • DCL（Data Control Language）：授權控制
💡
DBMS 主要組件：Database Interface → Semantic Data Controller → Query Processor → Transaction Manager → Concurrency Control Scheduler → Recovery Manager → Data Manager → Catalog Manager
CHAPTER 02

Entity-Relationship (ER) 模型

⭐
本章是重點中的重點！ 期中考有大量畫 ERD 的題目（Exercise #1~#6），要練熟所有概念的符號畫法。

🎨ER 圖常用符號總整理

符號代表說明
矩形 □Entity Type強實體型態（有自己的 Key）
雙矩形 ▣Weak Entity Type沒有自己的 Key，依賴 Owner
菱形 ◇Association Relationship Type一般關聯關係
雙菱形 ◈Identifying Relationship弱實體的識別關係
橢圓 ○Attribute一般屬性（single-valued）
雙橢圓 ⊙Multi-valued Attribute多值屬性（如電話、電郵）
虛線橢圓Derived Attribute可由其他屬性算出（如年齡）
底線屬性Key Attribute主鍵屬性
虛線底線屬性Partial Key弱實體的部分鍵
IS-A 三角Specialization繼承關係（超類→子類）

📦Entity 相關概念

  • Entity：真實世界中可辨識的物體（人、公司、課程…）
  • Entity Type：具相同屬性的 Entity 集合（如 EMPLOYEE）
  • Weak Entity Type：沒有自己的 Key Attribute，需靠 Owner 實體 + Partial Key 識別
⚠️
Weak Entity 一定有 Total Participation（完全參與）在識別關係中

🔑Key 種類

  • Key Attribute：可唯一識別每個 Entity 的屬性
  • Candidate Key：所有可能的 Key（可能有多個）
  • Primary Key：從 Candidate Key 中選定的主鍵
  • Secondary Key：未被選為 Primary Key 的其他 Candidate Key
  • Partial Key：弱實體的部分識別屬性（虛線底線）

🏷️Attribute 類型

Simple vs Composite
  • Simple（原子屬性）：不可再分割（如 SSN）
  • Composite（複合屬性）：可分解成子屬性（如 Address → Street, City, State）
Single-valued vs Multi-valued
  • Single-valued：每個 Entity 只有一個值（如 Name）
  • Multi-valued：可有多個值（如 Phone, Email）→ 畫雙橢圓
Stored vs Derived
  • Stored：直接儲存的屬性（如 BirthDate）
  • Derived：可從其他屬性算出（如 Age）→ 畫虛線橢圓
Null Values
  • Not Applicable：此屬性不適用於此 Entity
  • Unknown：不知道此屬性的值

🔗Relationship Type 概念

Degree（度數）
  • Unary（一元）：同一 Entity Type 與自己關聯
  • Binary（二元）：兩個 Entity Type（最常見）
  • Ternary（三元）：三個 Entity Type
Cardinality（基數限制）
  • 1:1：一對一
  • 1:M：一對多
  • M:M：多對多
  • 用 (min, max) 標記在各端
💡
如何看 (min, max)：某 Entity 最少參與幾次 / 最多幾次。例如 (1,1) = 剛好參與一次；(0,N) = 可以不參與，也可以參與多次。

🌳Specialization（泛化/特化）關係

Disjointness Constraint（排斥性）
  • Disjoint（D）：每個 superclass entity 最多屬於一個 subclass
  • Overlapping（O）：可同時屬於多個 subclass
Completeness Constraint（完整性）
  • Total（T）：每個 superclass entity 必屬於某個 subclass
  • Partial（P）：superclass entity 可以不屬於任何 subclass
組合標記
(D, P) = Disjoint + Partial Coverage
(D, T) = Disjoint + Total Coverage
(O, T) = Overlapping + Total Coverage
(O, P) = Overlapping + Partial Coverage

🔏Relationship Type 的 Key

  • 如果員工只能在同一個部門工作一次：Key = Employee.SSN
  • 如果員工可重複在同部門工作（需記歷史）：Key = Employee.SSN + StartingDate
  • 如果學生不能重複修同一門課：Key = Student.SSN + Course.C-No
  • 如果學生可重複修課（如補修）：Key = Student.SSN + Course.C-No + Term
CHAPTER 03

關聯式資料模型 (Relational Data Model)

📊基本概念

  • Relation：二維表格
  • Tuple（row）：一筆資料（一列）
  • Attribute（column）：一個欄位（一行）
  • Degree：屬性的數量（欄數）
  • Cardinality：Tuple 的數量（列數）
  • Domain：屬性的合法值域

📐Relation 的特性

  • 每個 cell 只能有 single value（原子值）
  • 每欄名稱唯一
  • 同欄的值都來自同一個 Domain
  • 無重複的 Tuple（每列都不同）
  • 欄的順序不重要
  • 列的順序不重要

🔑Primary Key & Foreign Key

Primary Key

唯一識別每個 Tuple 的屬性（或屬性組合）。不可為 NULL（Entity Integrity）。

Foreign Key

引用另一個 Relation 的 Primary Key。其值必須存在於被引用的 Relation 中，或為 NULL（Referential Integrity）。

⚖️四種 Constraint

Constraint說明違反範圍
Domain Constraint值必須來自屬性的 Domain單一屬性
Key ConstraintRelation 中不可有重複 Tuple單一 Relation
Entity IntegrityPrimary Key 不可為 NULL單一 Relation
Referential IntegrityForeign Key 值必須存在於被引用的 PK，或為 NULL兩個 Relation 之間

🛠️修改操作 (Modification Operations)

INSERT

可違反所有四種 Constraint

DELETE

只能違反 Referential Integrity（其他 Tuple 有 FK 指向被刪的）

處理選項：① Reject ② Cascade 刪除 ③ 設為 NULL

UPDATE

非 PK/FK 欄位 → 只能違反 Domain Constraint
更新 PK → 類似 Delete + Insert
更新 FK → 要確認新值存在

CHAPTER 04

ER → 關聯式模型 轉換規則

⭐
轉換規則（Rule 1~8）是超級重要的考題！要記住每個 Rule 對應什麼情況，以及如何決定 Primary Key 和 Foreign Key。

1️⃣Rule 1：Regular Entity Type（強實體）

  • 建立一個 Relation，包含所有 single-valued, simple 屬性（composite 屬性取其 components）
  • Primary Key = Entity Type 的 Key Attribute
  • Multi-valued 屬性另外用 Rule 8 處理
Student(SSN, FName, LName, Address)

2️⃣Rule 2：Weak Entity Type（弱實體）

  • 建立 Relation，包含所有 single-valued, simple 屬性
  • 複製 Owner Entity 的 PK 作為 FK
  • Primary Key = Owner Entity PK + Partial Key（部分鍵）
Dependent(SSN, Name, Gender, Relation)
-- SSN 是 FK，引用 Instructor

3️⃣Rule 3：1:1 Association Relationship

  • 選擇一個 Entity（設為 S），將另一個 Entity（T）的 PK 加進 S 的 Relation 作為 FK
  • 把 Relationship 的 single-valued 屬性也加到 S
  • FK 放哪邊都可以，通常放在 Total Participation（完全參與）那邊
-- 選項一：
Department(DNumber, DName, MgrSSN, StartDate)
Manager(SSN, Salary)  -- MgrSSN 是 FK → Manager

-- 選項二：
Department(DNumber, DName)
Manager(SSN, Salary, DNumberManaged, StartDate)  -- DNumberManaged 是 FK → Department

4️⃣Rule 4：1:M Association Relationship

  • 將 「1」那端 Entity 的 PK 複製到 「M」那端的 Relation 當 FK
  • Relationship 的屬性也加到「M」那端
Department(DNumber, DName)
Employee(SSN, EName, DNumber, StartDate)
-- DNumber 是 FK → Department（因為 Employee 是 M 那端）

5️⃣Rule 5：M:M Association Relationship

  • 新建一個 Relation 代表此 Relationship
  • 把兩端 Entity 的 PK 都作為 FK 加入新 Relation
  • 新 Relation 的 PK = 兩個 FK 的組合（加上 Relationship 的 key attributes）
  • Relationship 的屬性也加入
Employee(SSN, EName)
Project(PNumber, PName)
Works-On(SSN, PNumber, Hours)
-- SSN 和 PNumber 都是 FK，合起來是 PK

6️⃣Rule 6：N-ary Relationship（N > 2）

  • 新建一個 Relation
  • 把所有參與 Entity 的 PK 作為 FK 加入
  • PK = 所有 FK 的組合（但如果某端 maximal cardinality = 1，那端的 FK 不用放進 PK）
Work-on(S-SSN, A-SSN, Title)
-- 如果 Student 端 maximal cardinality = 1，PK = (A-SSN, Title)
-- 否則 PK = (S-SSN, A-SSN, Title)

7️⃣Rule 7：Specialization（IS-A 關係）→ 四種選項

選項作法適合情況
Option A Superclass + 每個 Subclass 各有一個 Relation，Subclass 複製 Superclass PK 作 FK 任何情況（最通用）
Option B 只有 Subclass 有 Relation，Superclass 屬性複製到每個 Subclass，刪除 Superclass Relation Disjoint + Total（不能有只屬於 Superclass 的 entity）
Option C-1 只有 Superclass 有 Relation，加入 Type 屬性（表示屬於哪個 Subclass），刪除 Subclass Relation Disjoint（排斥）子類
Option C-2 只有 Superclass 有 Relation，加入 m 個 Boolean 屬性（t1…tm） Overlapping（重疊）子類
⚠️
Option C-1 的 Type 屬性：若 Completeness = Total → Type 不可為 NULL；若 Partial → 可為 NULL

8️⃣Rule 8：Multi-valued Attribute（多值屬性）

  • 為此屬性建立一個新的 Relation
  • 包含：多值屬性 + 原 Entity 的 PK（作為 FK）
  • 新 Relation 的 PK = 多值屬性 + FK 的組合
Student(SSN, ..., Address)
Student-Phone(SSN, Phone)  -- SSN 是 FK，PK 是 (SSN, Phone)
CHAPTER 05

正規化 (Normalization)

🎯正規化的目標

  • 消除 Data Redundancy（資料重複）
  • 避免 Modification Anomalies（修改異常）：
    • Insertion Anomaly：新增資料時發生問題
    • Deletion Anomaly：刪除資料時意外損失其他資料
    • Update Anomaly：更新一筆資料但忘記更新其他重複的

→函數相依 (Functional Dependency, FD)

定義
X → Y 表示「X functionally determines Y」，即當兩個 tuple 的 X 值相同，Y 值一定也相同。
X = 左手邊（Determinant）；Y = 右手邊
Armstrong Axioms（推導規則） 
Reflexivity:   若 Y ⊆ X, 則 X → Y
Augmentation:  若 X → Y, 則 X,Z → Y,Z
Transitive:    若 X → Y 且 Y → Z, 則 X → Z
Decomposition: 若 X → Y,Z, 則 X → Y 且 X → Z
Union:         若 X → Y 且 X → Z, 則 X → Y,Z
Pseudotransitive: 若 X → Y 且 W,Y → Z, 則 W,X → Z

📊各階正規化一覽

1NF → 消除多值屬性和複合屬性，所有值都是原子值（atomic）
2NF → 在 1NF 基礎上，消除Partial FD（部分相依）：非 key 屬性必須完全相依於整個 Primary Key
3NF → 在 2NF 基礎上，消除Transitive FD（遞移相依）：非 key 屬性不能傳遞相依於 Primary Key
BCNF → 比 3NF 更嚴：所有屬性（包括 key 屬性）都不能傳遞相依於 PK

🧩Full vs Partial FD

Full FD（完全函數相依）
X → Y 且 X 的任何 proper subset 都無法 determine Y

例：SSN, P# → Hours 是 Full FD（光靠 SSN 或 P# 都不夠）

Partial FD（部分函數相依）
X → Y 且 X 的某個 proper subset 也能 determine Y

例：SSN, P# → PName 是 Partial FD（因為 P# → PName 就夠了）

🔄Transitive FD（遞移函數相依）

定義
如果 X → Z，且存在 Z 滿足：
(1) Z 不是 candidate key，也不是 PK 的 subset
(2) X → Z 且 Z → Y 都成立
→ 則 X → Y 是 Transitive FD

例：Works-For(SSN, D#, DPhone)，其中 SSN → D# 且 D# → DPhone
→ SSN → DPhone 是 Transitive FD（通過 D# 傳遞）

⚡2NF 正規化步驟

⚠️
如果 Relation 的 PK 只有一個屬性（非複合主鍵），它一定自動在 2NF！只有複合主鍵才需要考慮 2NF。
步驟
1. 找出所有 Partial FD（非 key 屬性相依於 PK 的某個 proper subset）
2. 對每個 Partial FD，建立新的 Relation，把 partial key 的 determinant 當 PK
3. 原 Relation 保留完整 PK + 完全相依的屬性，移除 partial dependency 的屬性 
Works-on(SSN, P#, Hours, PName)
-- P# → PName 是 Partial FD

2NF 後：
Works-On(SSN, P#, Hours)   -- PK = (SSN, P#)
Project(P#, PName)         -- PK = P#

⚡3NF 正規化步驟

步驟
1. 找出所有 Transitive FD
2. 把中間的 determinant（Z）提出來建立新 Relation，Z 作為新 Relation 的 PK
3. 原 Relation 保留 X 和 Z（Z 變成 FK），移除 Y 
Works-For(SSN, D#, DPhone)
-- SSN → D# → DPhone 是 Transitive FD

3NF 後：
Works-For(SSN, D#)               -- D# 是 FK → Department-Phone
Department-Phone(D#, DPhone)     -- PK = D#

⚡BCNF 正規化

定義 vs 3NF 的差別
3NF：只有非 key 屬性不能 transitively depend on PK
BCNF：所有屬性（包括 key 屬性本身）都不能 transitively depend on PK 
-- 例：R(A, B, C)，FDs: A,B → C  和  C → B
-- 主鍵 A,B。但 C → B 中，C 不是 candidate key → 違反 BCNF

BCNF 後：
R1(C, B)   -- PK = C
R2(A, C)   -- PK = A（C 是 FK）
⚠️
BCNF 分解後要小心：某些 BCNF 分解會產生 Spurious Tuples（假資料）！要確認用正確的 decomposition。

🎓經典範例：Course-Taking 完整流程

原始 Relation
Course-Taking(S-SSN, S-Name, C#, C-Name, I-SSN, I-Name, Grade)
PK = (S-SSN, C#)
已知 FDs： 
S-SSN, C# → S-Name, C-Name, I-SSN, I-Name, Grade（全部）
S-SSN → S-Name
C# → C-Name, I-SSN
I-SSN → I-Name
1NF ✅（所有值都是原子的）
2NF 正規化（消除 Partial FD）：
S-SSN → S-Name 是 Partial FD（只靠 S-SSN 就決定了）
C# → C-Name, I-SSN, I-Name 是 Partial FD 
Student(S-SSN, S-Name)
Course-Taking(S-SSN, C#, Grade)
Course-Teaching(C#, C-Name, I-SSN, I-Name)
3NF 正規化（消除 Transitive FD）：
在 Course-Teaching 中：C# → I-SSN → I-Name 是 Transitive FD 
Course-Teaching(C#, C-Name, I-SSN)   -- I-SSN 是 FK
Instructor(I-SSN, I-Name)
CHAPTER 06

考古題類型整理 & 解題技巧

📝ERD 繪製題型 — 解題步驟

標準解題流程
1. 列出所有 Entity Type：名詞通常是 entity
2. 找屬性：看哪些屬性有唯一性（→ Key）、多值性（→ multi-valued）
3. 找 Relationship：動詞通常是 relationship（訂購、參與、管理…）
4. 標記 Cardinality：看題目「一位 X 最多/最少對應幾個 Y」
5. 找 Weak Entity：有沒有「屬於某個 entity，沒有獨立的 key」
6. 找 Specialization：有沒有「可分類」的 entity（IS-A 關係）

🩺常見 ERD 題目整理

題目重點 Entity特別注意
Exercise #1 外送餐食系統 Employee, Store, FoodItem, Order, Deposit FoodItem 是 Weak Entity（對同一家 Store，fName 唯一；但不同 Store 可能重複）
Exercise #2 課程資料庫 Course, Teacher, Student, Item Item 是 Weak Entity（對同一 Course，iName 唯一）；M:M 的 Takes, Teaches
Exercise #3 拍賣網站 Member, Merchandise, Category, Bid Merchandise 是 Weak Entity（seqNo 在同一 Member 下唯一）；Category 有 IsParent 自我關係
Exercise #4 醫院掛號 Department, Doctor, DiagnosisTime, Patient DiagnosisTime 是 Weak Entity（開始時間在同一 Doctor 下唯一）
Exercise #5 擴充掛號+住院 +ChiefDoc, AttendingDoc, Resident, Inpatient, Room Doctor IS-A 特化（Disjoint + Total）；Inpatient IS-A Patient；Occupies Room
Exercise #6 ERD 工具軟體 EntityType, WeakEntityType, RelationshipType, Attribute 這題很特別——要用 ERD 描述 ERD 本身！複合屬性的 Composes 關係

⚙️Relational Transformation 題型

💡
拿到 ERD，按 Rule 1→2→3→4→5→6→7→8 依序處理，不要跳過任何一種 Entity/Relationship Type。
常見陷阱：
  • Weak Entity 的 PK 一定是 Owner PK + Partial Key，不要忘記
  • 1:1 的 FK 放哪邊都可以，但要說明你的選擇
  • Multi-valued 屬性一定要另建 Relation（Rule 8）
  • Specialization Option B 只能用在 Disjoint 子類（不然一個 entity 可能要出現在多個 relation）
  • BCNF 的例子：注意 Spurious Tuples 問題

📐Normalization 題型解題步驟

萬用流程
1. 確認 Primary Key
2. 列出所有 FD
3. 1NF 檢查：有沒有多值或複合屬性？
4. 2NF 檢查：PK 是不是複合的？有沒有 Partial FD？
    → 有 → 分解，把 partial determinant 拉出去建新表
5. 3NF 檢查：有沒有 Transitive FD（non-key → non-key）？
    → 有 → 分解，把中間的 Z 拉出去建新表
6. BCNF 檢查：有沒有 determinant 不是 candidate key？
    → 有 → 分解，注意 lossless decomposition

🧠容易混淆的概念整理

概念容易搞混的點
Weak Entity vs Strong Entity Weak = 沒有自己的 Key（不是「沒有屬性」）；一定要有 Identifying Relationship
Partial Key vs Primary Key Partial Key 本身不夠識別，要加上 Owner 的 PK 才夠
Partial FD vs Transitive FD Partial FD = 左邊是 PK 的子集；Transitive FD = 中間有 non-key 屬性
3NF vs BCNF 3NF 允許 non-key 屬性 determine another non-key，只要不 transitively depend on PK；BCNF 連這個都不允許
Logical vs Physical Data Independence Logical = 改 Conceptual 不影響 External；Physical = 改 Internal 不影響 Conceptual
Domain vs Schema Domain = 屬性的合法值的集合；Schema = 資料庫的結構描述（Metadata）
